# 📘 BLE 学习笔记：Bluetooth Device Name & GATT 架构

------

## 一、Bluetooth Device Name（BD Name）概览

在 BLE 中，**设备名并非只有一个**，而是存在于两个不同阶段、两个不同机制中。

### 1️⃣ 广播设备名（Advertised Device Name）

- 出现阶段：**未连接 / 扫描时**
- 载体：**Advertising Data**
- 长度限制：**31 字节（可连接时实际可用 28 字节）**
- 作用：
  - 显示在扫描列表
  - 帮助用户快速识别设备

------

### 2️⃣ 完整设备名（Full Device Name）

- 出现阶段：**已连接**
- 载体：**GATT → GAP Service → Device Name Characteristic**
- 最大长度：**248 字节**
- 作用：
  - 作为设备“正式身份”
  - 常被手机系统或 App 缓存

------

✅ **关键结论**

> 广播名 ≠ 完整名
> 扫描阶段与连接阶段使用的是**不同数据通道**

------

## 二、默认设备名的来源（DIALOG-TMPL）

SDK6 默认行为：

- 在 `user_config.h` 中定义：

```c
#define USER_DEVICE_NAME "DIALOG-TMPL"
```

- SDK 在 `default_advertise_operation()` 中：
  - 自动构造 Advertising Data
  - 自动加入：
    - Flags
    - Device Name
    - 其他 GAP 参数

因此扫描到的默认名称为 **DIALOG-TMPL**

------

## 三、最简单的改名方式（编译期）

### 修改位置

```c
// user_config.h
#define USER_DEVICE_NAME "My Product"
```

### 生效流程

1. Build
2. 下载到芯片
3. 重新扫描

### iOS 注意事项

- iOS 会缓存广播名
- 必须 **连接 → 断开 → 再扫描**
- 否则可能仍显示旧名称

------

## 四、为什么可以使用 Emoji（Unicode 原理）

- BLE 设备名本质是 **字节数组**
- 不关心字符语义，只关心字节内容
- UTF-8 Emoji 本质是多字节编码

例如 😎：

```
F0 9F 98 8E
#define USER_DEVICE_NAME "\xF0\x9F\x98\x8E"
```

⚠️ 注意：

- 编码必须是 UTF-8
- 广播总长度不能超限

------

## 五、BLE Advertising Data 基础

### 1️⃣ 广播数据基本结构

每个 Advertising Data Segment 格式如下：

| 字段   | 长度   |
| ------ | ------ |
| Length | 1 字节 |
| Type   | 1 字节 |
| Data   | N 字节 |

示意：

```
[ Length ][ Type ][ Data... ]
```

------

### 2️⃣ 完整设备名的 AD Type

```c
GAP_AD_TYPE_COMPLETE_NAME = 0x09
```

含义：**Complete Local Name**

------

### 3️⃣ 广播数据长度限制

| 项目          | 占用    |
| ------------- | ------- |
| Flags（必须） | 3 字节  |
| 剩余可用      | 28 字节 |

- Flags 类型：`GAP_AD_TYPE_FLAGS`
- SDK6 会自动添加 Flags

## 六、为什么需要“运行时”修改设备名？

典型应用场景：

- 名字中包含 **序列号**
- 名字中包含 **芯片 ID**
- 同一固件烧录多台设备

例如：

```
MySensor_001
MySensor_002
```

👉 编译期宏无法满足动态需求

## 九、手动构造广播数据（示例）

```c
uint8_t adv_data[] = {
  0x05,                        // Length
  GAP_AD_TYPE_COMPLETE_NAME,   // Type = 0x09
  0xF0, 0x9F, 0x90, 0x9E        // 🐞 Emoji
};
```

| 字段      | 含义               |
| --------- | ------------------ |
| 0x05      | 段长度（不含自身） |
| 0x09      | 完整设备名         |
| 后 4 字节 | UTF-8 名称         |

写入广播命令：

```c
cmd->info.host.adv_data_len = sizeof(adv_data);
memcpy(&cmd->info.host.adv_data, adv_data, sizeof(adv_data));
```

## 十、iOS 显示旧名字的原因

原因：
👉 **只修改了广播名，没有修改完整设备名**

- 扫描列表：来自 Advertising Data
- 连接后信息页：来自 GAP Service

------

## 十一、同步修改完整设备名（关键）

```c
device_info.dev_name.length = sizeof(adv_data) - 2;
memcpy(device_info.dev_name.name,
       &adv_data[2],
       device_info.dev_name.length);
```

原理：

- 跳过 Length 和 Type
- 仅复制真实名称数据

结果：

- 扫描看到 🐞
- 连接后也看到 🐞

------

## 十二、BLE Explorer 显示差异的根源

不同 App 行为不一致：

- 有的显示广播名
- 有的显示完整名
- 有的缓存
- 有的不标明来源

👉 常见问题来源是 **App 行为差异，而非固件错误**

## 十三、Flags 的补充说明（易忽略）

- Flags 仅在 **可连接广播** 中是强制的
- 不可连接广播可用完整 31 字节

| 模式     | 可用字节 |
| -------- | -------- |
| 可连接   | 28       |
| 不可连接 | 31       |

------

## 十四、GATT 是什么？（核心概念）

**GATT（Generic Attribute Profile）**
是 BLE **连接后**的数据通信协议。

一句话总结：

> **BLE 连接后，所有数据交互都通过 GATT 的服务和特征完成。**

------

## 十五、GATT 的结构模型

```
GATT Database
│
├── Service
│    ├── Characteristic
│    │     └── Value + Descriptor
│
└── Service
     └── Characteristic
```

- Service：功能集合
- Characteristic：可读写的数据点

------

## 十六、常见的两个标准 GATT 服务

### 1️⃣ Device Information Service（UUID: 0x180A）

提供设备基本信息：

- Manufacturer Name
- Model Number
- Serial Number
- Firmware / Hardware Revision

------

### 2️⃣ GAP Service（UUID: 0x1800）

提供设备身份信息：

- Device Name（完整设备名）
- Appearance
- Preferred Connection Parameters

------

## 十七、Advertising vs GATT 对比总结

| 阶段 | 数据来源         | 用途               |
| ---- | ---------------- | ------------------ |
| 扫描 | Advertising Data | 显示名称、UUID     |
| 连接 | GATT Database    | 设备信息、数据交互 |

------

## 🎯 最终核心认知（必记）

- **广播名用于“被发现”**
- **完整名用于“被识别”**
- **扫描靠 Advertising**
- **通信靠 GATT**
- **名字存在两个世界，必须分别管理**
