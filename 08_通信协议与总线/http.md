## 一、HTTP报文结构

#### 1.请求报文

**（1）.请求行**

- **方法 (Method)：**表明意图。

  - `GET`：获取资源（只读）。

  - `POST`：提交数据（通常用于创建新资源，如上传文件、发送 JSON 数据）。
  - `PUT`：更新资源（全量替换）。
  - `DELETE`：删除资源。

- **URL (统一资源定位符)：**请求的具体路径，例如 `/api/v1/sensors/temp`。

- **协议版本：** 通常是 `HTTP/1.1`。

- *(示例：`POST /api/v1/sensors HTTP/1.1\r\n`)*

**（2）.请求头 (Headers)**

- `Host`: 目标服务器的域名（必填）。
- `Content-Type`: 告诉服务器，我的请求体是什么格式（例如 `application/json`）。
- `Content-Length`: 告诉服务器，我的请求体有多长（字节数），这决定了服务器何时停止读取这一个报文。

- **空行 (Blank Line)：** 仅仅是一个回车换行符 `\r\n`。它标志着 Header 的结束和 Body 的开始。
- **请求体 (Body)：** 实际要发送的数据内容。GET 请求通常没有 Body，而 POST/PUT 请求会有（比如一串 JSON 字符串）。

**请求报文原始长相示例：**

```
POST /update HTTP/1.1
Host: api.example.com
Content-Type: application/json
Content-Length: 27

{"temperature": 25, "id": 1}
```

### 2. 响应报文 (Response)

服务器处理完请求后，返回给客户端的结果。

- **状态行 (Status Line)：** 包含协议版本、状态码和状态描述。
  - *(示例：`HTTP/1.1 200 OK\r\n`)*
- **响应头 (Headers)：** 类似请求头，服务器通过它告诉客户端一些信息。
  - `Content-Type`: 告诉客户端返回的数据是什么格式（例如 `text/html` 或 `application/json`）。
- **空行 (Blank Line)：** `\r\n`，标志着响应头结束。
- **响应体 (Body)：** 服务器实际返回的数据（比如网页的 HTML 源码，或者接口返回的 JSON 结果）。

**响应报文原始长相示例：**

```
HTTP/1.1 200 OK
Content-Type: application/json
Content-Length: 35

{"status": "success", "message": ""}
```

## 二、 状态码分类 (Status Codes)

- **1xx (信息性状态码)：** 表示请求已被接收，正在继续处理。（在实际开发中较少直接接触）。
- **2xx (成功状态码)：** 表示请求已成功被服务器接收、理解并接受。
  - **`200 OK`：** 最常见，请求成功。
  - **`201 Created`：** 通常用于 POST 请求，表示资源成功在服务器端被创建。
- **3xx (重定向状态码)：** 表示需要客户端采取进一步的操作才能完成请求。
  - **`301 Moved Permanently`：** 永久重定向（资源换了新网址）。
  - **`302 Found`：** 临时重定向。
- **4xx (客户端错误状态码)：** **这是客户端（你写的代码）的锅。** 服务器理解了请求，但拒绝执行或发现请求有语法错误。
  - **`400 Bad Request`：** 请求报文有语法错误（比如 JSON 格式没写对，或者缺少了必填的 Header）。
  - **`401 Unauthorized`：** 未授权（通常是没带 Token 或者密码错了）。
  - **`403 Forbidden`：** 身份验证通过了，但是你没有权限访问这个特定资源。
  - **`404 Not Found`：** 请求的 URL 在服务器上不存在。
- **5xx (服务器错误状态码)：** **这是服务器的锅。** 客户端发送的请求完全正确，但服务器内部在处理时崩溃了或遇到了不可预期的错误。
  - **`500 Internal Server Error`：** 服务器内部代码报错、崩溃。
  - **`502 Bad Gateway`：** 网关错误（通常是后端的真实服务器挂了，导致前置的代理服务器拿不到数据）。

#### 三、记忆机制

- **Cookie（客户端的记忆）：**
  - **原理：** 你第一次登录成功后，服务器会在响应头里塞一个 `Set-Cookie: session_id=12345`。你的浏览器（或设备）会把这串数据存下来。下次再发请求时，设备会自动在请求头里带上 `Cookie: session_id=12345`。
- **Session（服务器的记忆）：**
  - **原理：** Cookie 存在客户端，容易被篡改。所以服务器在自己内存（或数据库）里建一个Session。账本里记录 `12345` 对应的是“用户已登录”。客户端只需要拿着 `12345` 这个号码牌（通常通过 Cookie 传输）来对暗号即可。
- **Token (如 JWT，现代 IoT 常用)：**
  - 在物联网设备与云端通信时，更常用的是 Token。服务器用密钥对用户信息进行签名生成一串字符串（Token）发给设备。设备每次请求都在 Header 中带上 `Authorization: Bearer <Token>`。服务器不需要查账本，只要用算法验证签名有效，就放行。
